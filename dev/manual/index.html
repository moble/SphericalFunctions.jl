Function calculators

Typically, when calculating special functions, we will use recursion relations along with some coefficients ‚Äî which frequently requires significant setup processing. That processing can be cached, so that the calculations themselves consist primarily of memory accesses and simple arithmetic.

Similarly, some computations require a certain amount of "workspace" ‚Äî significant amounts of memory that are needed for the computation, but are not part of the core result (and don't even need to be returned). Rather than allocating this memory on each call to the computation, we can allocate once and pass the workspace in as an argument.

These two patterns ‚Äî pre-computed coefficients and pre-allocated workspaces ‚Äî will appear as possible call signatures in each of the following functions. Note that we also have helper functions like Dprep, Dstorage, etc., which construct the necessary coefficients and/or workspaces, as well as the actual results for in-place computations, to make it easier to call the corresponding calculator functions.

Note that these functions impose assumptions about the ordering of arrays as vectors. For example, the series of Wigner $ùîá$ matrices will be stored as a single vector. And for particular values of $(‚Ñì, n, m)$, the element $ùîáÀ°‚Çô,‚Çò$ will have to be accessed as some linearly indexed element of that vector. These formats, along with utility functions for accessing particular elements, are described here.

The fundamental algorithm is the $H$ recursion, which is the core computation needed for Wigner's $d$ and $ùîá$ matrices, and the spin-weighted spherical harmonics ${}_{s}Y_{\ell,m}$, as well as map2salm functions.

This computation forms the basis for computing Wigner's $d$ and $ùîá$ matrices via d! and D!, the spin-weighted spherical harmonics via Y!, and for transforming from values of spin-weighted spherical functions evaluated on a grid to the corresponding mode weights via map2salm.

Due to symmetries, we only need to compute ~1/4 of the elements of this matrix, so only those elements with $m ‚â• |m‚Ä≤|$ are computed. The relevant indices of the H vector are computed based on the Hindex function ‚Äî which defaults to WignerHindex, but could reasonably be WignerDindex if the input H vector contains all valid indices. However, it is assumed that the storage scheme used for H is such that the successive $m$ values are located in successive elements.

Compute the "fully normalized" Associated Legendre Functions up to some maximum n value nmax.

Transformation

The functions above ‚Äî especially Y! ‚Äî allow us to evaluate mode weights (often denoted salm) of a spin-weighted spherical-harmonic expansion as a function of a particular point (often denoted map). This process is frequently referred to as "synthesis", and is also called salm2map.

We can also perform the reverse process of going from the function evaluated on some kind of grid, back to the corresponding mode weights ‚Äî also called "analysis" or map2salm.

This can be useful, for example, when processing multiple fields with different spin weights; the function could be evaluated on points appropriate for the lowest value of $|s|$, and therefore could also be used to solve for fields of all other spin weights.

Note that in-place operation is possible for this type when the length of the input RŒ∏œï is equal to the number of modes given s and ‚Ñì‚Çò‚Çê‚Çì ‚Äî and is the default behavior when possible. See SSHT for description of in-place operation.

This method is typically better than other current implementations for $‚Ñì‚Çò‚Çê‚Çì ‚â≤ 24$, both in terms of speed and accuracy. However, this advantage quickly falls away. A warning will be issued if ‚Ñì‚Çò‚Çê‚Çì is greater than about 64, because this method is not likely to be the most efficient or most accurate choice.

SphericalFunctions.SSHTRS ‚Äî Method
SSHTRS(s, ‚Ñì‚Çò‚Çê‚Çì; kwargs...)

Construct a SSHTRS object directly. This may also be achieved by calling the main SSHT function with the same keywords, along with method="RS".

This object uses the algorithm described in this paper by Reinecke and Seljebotn.

The basic floating-point number type may be adjusted with the keyword argument T, which defaults to Float64.

The SSHs are evaluated on a series of "rings" at constant colatitude. Their locations are specified by the Œ∏ keyword argument, which defaults to fejer1_rings(2‚Ñì‚Çò‚Çê‚Çì+1, T). If this is changed, the user should also provide the corresponding quadrature_weights argument ‚Äî the default being fejer1(length(Œ∏), T).

On each of these rings, an FFT is performed. To reach the band limit of $m = ¬± ‚Ñì‚Çò‚Çê‚Çì$, the number of points along each ring must therefore be at least $2‚Ñì‚Çò‚Çê‚Çì+1$, but may be greater. For example, if $2‚Ñì‚Çò‚Çê‚Çì+1$ does not factorize neatly into a product of small primes, it may be preferable to use $2‚Ñì‚Çò‚Çê‚Çì+2$ points along each ring. (In that case, whenever ‚Ñì‚Çò‚Çê‚Çì is 1 less than a power of 2, the number of points will be exactly a power of 2, which is usually particularly efficient.) The number of points on each ring can be modified independently, if given as a vector with the same length as Œ∏, or as a single number which is assumed to be the same for all rings.

Whenever T is either Float64 or Float32, the keyword arguments plan_fft_flags and plan_fft_timelimit may also be useful for obtaining more efficient FFTs. They default to FFTW.ESTIMATE and Inf, respectively. They are passed to AbstractFFTs.plan_fft.

SphericalFunctions.sorted_rings ‚Äî Method
sorted_rings(s, ‚Ñì‚Çò‚Çê‚Çì, [T=Float64])

Compute locations of a series of rings labelled by $j ‚àà |s|:‚Ñì‚Çò‚Çê‚Çì$ (analogous to $‚Ñì$), where each ring will contain $k = 2j+1$ (analogous to $m$) pixels distributed evenly around the ring. These rings are then sorted, so that the ring with the most pixels ($j = ‚Ñì‚Çò‚Çê‚Çì$) is closest to the equator, and the next-largest ring is placed just above or below the equator (depending on the sign of $s$), the next just below or above, and so on. This is generally a fairly good first guess when minimizing the condition number of matrices used to solve for mode weights from function values. In particular, I use this to initialize the Minimal algorithm, which is then fed into an optimizer to fine-tune the positions of the rings.

This function does not provide the individual pixels; it just provides the colatitude values of the rings on which the pixels will be placed. The pixels themselves are provided by sorted_ring_pixels.
